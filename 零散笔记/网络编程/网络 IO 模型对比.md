## 网络 I/O 模型对比：从阻塞到多路复用
### 1. 传统阻塞 I/O (Blocking I/O)

**模型描述**  
线程调用 `accept()` 等待新连接，或调用 `read()` 读取数据，若暂无数据，线程会被操作系统**挂起**，直到数据就绪才被唤醒继续执行。若需处理多个连接，必须为每个连接分配一个独立线程（或进程），每个线程在自己的 fd 上阻塞等待。

**典型特征**
- **阻塞点**：每个线程在其负责的单一 fd 上执行 `read()` / `accept()` 时挂起。
- **并发能力**：受限于线程数，一个线程只能服务一个连接。
- **资源消耗**：每个线程都有独立栈空间（通常 ≥ 1MB），大量连接导致内存膨胀，且 CPU 频繁进行上下文切换。
- **处理能力**：线程大多数时间处于阻塞状态，CPU 利用率低，无法应对 C10K 级别并发。

**适用场景**  
早期低并发场景、连接数少且交互简单的网络服务（如早期 Tomcat BIO 模式、Apache prefork）。

---

### 2. 单线程非阻塞轮询 (Non-blocking I/O with Busy Polling)

**模型描述**  
将所有的 socket 设置为**非阻塞模式**，单线程循环依次对每个 socket 调用 `read()` 或 `accept()` 等系统调用。若无数据，调用立即返回错误码（如 `EAGAIN`），线程继续轮询下一个 socket，永不阻塞在单个连接上。

**典型特征**
- **阻塞点**：无真正的阻塞等待，线程一直在循环执行系统调用（忙轮询）。
- **并发能力**：单线程即可管理多个连接，因为不会卡在某个连接上。
- **系统调用开销**：**极大**。每次轮询对每个 fd 都要进行一次系统调用（用户态 ↔ 内核态切换），即使 fd 无数据可读。
- **CPU 使用**：极高。无论连接是否有数据，线程都在全速轮询，造成严重 CPU 空转浪费。
- **实现复杂度**：程序需要自己维护所有 fd 列表及轮询逻辑。

**适用场景**  
仅作为理解 IO 多路复用的铺垫，实际**极少使用**，除非连接数极低且对延迟要求极端苛刻的特殊场景（如某些实时系统或驱动程序）。

---

### 3. IO 多路复用 (IO Multiplexing)

**模型描述**  
单线程通过一次系统调用（如 `select`、`poll`、`epoll_wait`、`kqueue`）将待监听的**所有 fd** 一次性提交给内核，然后**阻塞在这一个调用上**。内核会在其中**任意 fd** 就绪（可读、可写、异常）时唤醒线程，并返回所有就绪 fd 的列表。线程随后对这些就绪 fd 进行非阻塞读写，处理完毕后继续回到复用调用上等待。

**典型特征**
- **阻塞点**：线程阻塞在 `epoll_wait()` 等复用函数上，但同时等待所有 fd，实现“一对多”等待。
- **并发能力**：极强。单线程可高效管理数万甚至数十万连接，突破 C10K/C100K 问题。
- **系统调用开销**：极小。一次阻塞等待即可获得多个就绪事件，避免了对每个空闲 fd 的无效系统调用。
- **CPU 使用**：按需唤醒，绝无空转。线程仅在事件就绪时工作。
- **实现复杂度**：依赖操作系统提供的多路复用接口，配合非阻塞 I/O 和事件驱动编程模型（Reactor/Proactor）。

**适用场景**  
所有高并发网络服务，如 Nginx、Redis、Netty、Node.js、Go 的网络层等。

---

## 三种模型的核心对比表

| 特性                | 传统阻塞 I/O                   | 单线程非阻塞轮询              | IO 多路复用                      |
| ------------------- | ------------------------------ | ----------------------------- | -------------------------------- |
| **线程使用**        | 每个连接一个线程（或进程）     | 单线程管理多个连接            | 单线程（或少量线程）管理海量连接 |
| **阻塞行为**        | 阻塞在单个 fd 的 `read`/`accept` 上 | 永不阻塞，忙循环轮询          | 阻塞在一次 `epoll_wait` 等多路复用调用上，同时监听所有 fd |
| **等待机制**        | 被动挂起，单 fd 唤醒           | 主动反复查询，无等待          | 被动挂起，多 fd 唤醒             |
| **系统调用次数**    | 少（仅阻塞那次）               | 极多（每轮每 fd 一次）        | 极少（一次等待，批量返回）       |
| **CPU 利用效率**    | 低（切换开销大，大量空转等待） | 极低（空转轮询浪费 CPU）      | 高（精确事件驱动，无空转）       |
| **并发连接支持**    | 低（数百～数千）               | 中（受限于轮询开销）          | 极高（数万～百万）               |
| **实现难度**        | 简单                           | 繁琐（需自行轮询 + 状态维护） | 中等（需理解事件驱动模型）       |

---

### 总结

- **传统阻塞 I/O** 最直观简单，但“一个线程死等一个连接”的本质导致其并发扩展能力极差。
- **单线程非阻塞轮询** 尝试用“不断询问”解决阻塞问题，却引来了**高频用户态/内核态切换**和**CPU 空转**的高昂代价。
- **IO 多路复用** 通过 **“一次提交，批量等待，精准唤醒”** 的机制，将连接管理和事件分发统一到内核高效实现，使单线程得以驾驭海量连接，是现代高性能网络服务的基石。
- **单线程非阻塞轮询** 是在**用户**态轮询，**IO 多路复用** 是在**内核**态轮询（如select\poll）

## IO多路复用
> **select、poll、epoll** 都是 IO 多路复用的具体实现，它们都在解决同一个问题：**如何让一个线程同时高效地监听多个文件描述符的事件。**\
它们最核心的区别：**到底是内核在被动地“帮你全量轮询”，还是内核真正做到了“主动精准通知”。**

### 1. select 工作原理

**机制**：内核在用户态与内核态之间“来回拷贝 + 全量扫描”。

1. 用户程序构造一个 `fd_set` 位图（默认最大 1024 位），将感兴趣的 fd 置位。
2. 调用 `select()`，把整个位图**拷贝到内核空间**。
3. 内核开始**遍历整个位图**，检查每个 fd 是否有事件就绪。如果没有，就把当前线程挂在这个所有 fd 的等待队列上，进入休眠。
4. 当任意一个 fd 有事件到达时，内核唤醒线程，**再次遍历整个位图**，找到就绪的那些 fd，标记好。
5. `select` 返回，把整个位图**重新拷贝回用户空间**，用户再遍历一次找出就绪的 fd，未就绪的则需要重新写入位图。

**核心缺陷**：

-   **数量限制**：位图大小固定，通常只能监听 1024 个 fd（受 `FD_SETSIZE` 宏限制）。
-   **全量拷贝与扫描**：每次调用都要传入并拷贝整个集合，内核和用户态都要 `O(n)` 遍历，n 为最大 fd 值，非常低效。
-   **破坏性**：返回的 `fd_set` 是修改过的，下次循环必须重新构造。

**用我们的比喻**：select 就像你给了管家一份最多 1024 人的名册，每次问他“谁有消息？”，他都要把名册从头到尾看一遍，看完再把你那份名册涂改一下还给你，你自己还得再看一遍。费时费力，名册还不能超过 1024 人。

---

### 2. poll 工作原理

**机制**：本质上和 select 一样，也是“全量扫描”，但换了个更灵活的数据结构。

1. 用户构造一个 `pollfd` 的**动态数组**，每个元素描述一个 fd 和感兴趣的事件。
2. 调用 `poll()`，把整个数组拷贝到内核。
3. 内核**遍历整个数组**，挂起等待。
4. 事件到达后，内核再次遍历数组，设置每个 `pollfd` 的 `revents` 字段。
5. `poll` 返回，把数组拷贝回用户态，用户遍历数组找出 `revents` 非零的项。

**与 select 相比的改进**：

-   **没有数量限制**：数据结构是链表或数组，理论上 fd 数量只受系统限制，不再受 1024 的限制。

**依然存在的核心缺陷**：

-   **全量拷贝与扫描依然存在**：每次调用照样要拷贝整个数组，内核照样是 `O(n)` 的全量遍历。当连接数上万时，每一次 `poll` 调用本身的开销就极大，无法应对高并发。

**比喻**：管家不再用固定大小的名册了，改成不限人数的清单。但他每次还是要你抄一份给他，他还是从头到尾看一遍，再涂改后还给你。人一多，光看清单就累死了。

### 3. epoll 工作原理

**机制**：真正的“事件驱动”模型。内核主动通知，操作复杂度接近 `O(1)`。核心是三个函数和内核中的两个数据结构。

-   **内核数据结构**：
    -   **红黑树**：用于高效存储和管理用户注册的 fd（增删改）。
    -   **就绪链表**：一个双向链表，用于存放刚好有事件发生的 fd。

-   **工作流程**：
    1.  **`epoll_create`**：创建内核事件表（红黑树 + 就绪链表），返回一个 epoll 实例的 fd。
    2.  **`epoll_ctl`**：向 epoll 实例**注册、修改、删除**要监听的 fd 和事件类型。这只是一次系统调用，内核把这个 fd 挂到红黑树上，并给该 fd 的内核缓冲区设置一个回调函数：“当你数据就绪了，就调用这个函数，把你插到就绪链表上去”。
    3.  **`epoll_wait`**：进程阻塞在这里。内核检查**就绪链表**。如果非空，就**直接把就绪链表上的事件信息拷贝到用户空间**，然后返回。如果为空，就休眠，直到有 fd 就绪触发回调、把自己加到链表上，再唤醒进程。

**为什么 epoll 高效？**

-   **注册一次，终身受用**：fd 只在 `epoll_ctl` 时拷贝一次到内核，后续 `epoll_wait` 不需要再传成千上万个 fd，彻底消除了 `O(n)` 的全量拷贝。
-   **操作复杂度 `O(1)`**：`epoll_wait` 直接检查就绪链表，链表里有几个就绪事件，就返回几个，**没有全量扫描**。fd 再多，只要活跃的不多，返回就极快。
-   **回调驱动**：fd 数据到达时，内核通过中断和回调机制**主动**把自己挂到就绪链表，而不是等每次 `epoll_wait` 时才去被动扫一遍红黑树。

**比喻**：这次你干脆给了管家一个呼叫器。你提前用 `epoll_ctl` 告诉他：“帮我盯着这 10000 个信箱”。每个信箱上都装了个机关，只要一有信，就自动按亮呼叫器上一个灯（回调函数把自己插入就绪链表）。你只需要坐着，等 `epoll_wait` 这个总指示灯一亮，它直接告诉你“3号、500号、8000号信箱有信”，你直接去拿就行，根本不用管另外 9997 个空信箱。

### 4. 三者的核心区别总结表

| 特性 | **select** | **poll** | **epoll** |
| :--- | :--- | :--- | :--- |
| **数据结构** | 位图 `fd_set` | 动态数组 `pollfd` | 内核红黑树 + 就绪链表 |
| **最大连接数** | 默认1024，可编译修改但效率低 | 无硬性限制，仅受系统资源限 | 无硬性限制，可达数十万+ |
| **内核态工作方式** | 每次都全量轮询所有 fd | 每次都全量轮询所有 fd | **事件回调**，只关注就绪 fd |
| **操作复杂度** | **O(n)** | **O(n)** | **O(1)** (仅取决于就绪数) |
| **拷贝开销** | 每次调用都复制整个集合 | 每次调用都复制整个数组 | fd 注册一次，仅拷贝就绪事件 |
| **适用场景** | 极少量连接，古老兼容 | 中等连接数 | **所有高并发服务器首选** |

---

### 5. 触发模式：epoll 的独有特性

epoll 还支持两种事件触发模式：

-   **水平触发**：只要 fd 还有数据可读，每次 `epoll_wait` 都会通知你。这和 select/poll 的默认行为一样，编程更简单安全，但可能带来重复通知。
-   **边缘触发**：仅在 fd 的状态从“无数据”变为“有数据”的那一刻通知一次。这要求程序必须配合非阻塞 I/O，一次性把缓冲区读干净，效率最高，但编程难度也更高。这是 epoll 发挥极致性能的关键。

### 总结

“轮询需要主动从用户态切到内核态访问数据是否就绪”，select 和 poll 就是**内核替你在内核态做了一次低效的全量轮询**。而 epoll 彻底改变了这一点，实现了**内核在数据就绪后，通过回调机制主动把就绪事件通知给用户态**。这正是 epoll 成为 Linux 高并发基石的根本原因。

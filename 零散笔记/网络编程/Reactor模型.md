## Reactor 模型
> **Reactor 模型**，直译为反应堆模型，是 IO 多路复用在服务器架构上的设计落地。它把 **`epoll_wait` 等待事件 -> 事件到达 -> 分发给对应的处理函数** 这一整套流程，抽象成了一种事件驱动的编程模式。

简单说，**Reactor 模型 = IO 多路复用 + 事件分发机制**。

### 1. 为什么需要 Reactor 模型？

一个线程阻塞在 `epoll_wait()`，一旦就绪，它获得了哪些 fd 可读/可写了。但接下来呢？服务器需要：

-   如果是监听 fd 就绪，要调用 `accept` 建立新连接。
-   如果是客户端 fd 可读，要读取、解码、计算、编码、写回。
-   可能还有超时处理、异常关闭等。

如果把所有这些逻辑全部揉在一个大循环里，代码会极度混乱。Reactor 模型的核心思想就是：**将事件的检测和事件的处理分离开**。Reactor 只负责监听并分发事件，具体怎么处理，交给对应的 Handler（处理器）去做。

---

### 2. Reactor 模型的核心组件

-   **Reactor（反应堆）**：负责监听事件。它持有 IO 多路复用器（如 epoll），不断执行 `select/poll/epoll_wait`，当事件到达时，立即分发给对应的处理器。**它不关心具体业务逻辑。**
-   **Handler（处理器）**：处理具体事件的业务代码。比如 Acceptor 负责接收连接，IO Handler 负责读写和业务处理。
-   **Event Demultiplexer（事件多路分解器）**：这就是我们一直在说的 `select/poll/epoll`，它在 Reactor 内部被调用，负责等待一堆 fd 上的事件发生。

---

### 3. Reactor 的常见线程模型

Reactor 模式最巧妙的地方在于，它可以非常灵活地与线程结合，衍生出不同的并发模型。

#### 模型一：单线程 Reactor（Redis 早期采用）

所有 I/O 和业务处理都在**一个线程**里完成。

-   **流程**：Reactor 线程在 `epoll_wait` 上等待，新连接到来，由 Acceptor 处理，建立连接并注册到 epoll 中；已连接 fd 可读，由对应的 IO Handler 读取数据、执行业务逻辑、最后写回响应。**整个过程在一个线程里串行完成。**
-   **优点**：实现简单，没有线程间的争抢和锁的问题，对进程内数据修改天然安全。
-   **致命弱点**：**一个 CPU 密集型任务会阻塞整个 Reactor 线程**，导致所有连接的事件都得不到处理。这验证了我们之前讨论的“CPU 密集要交给线程池”的必要性。
-   **适用场景**：IO 密集且计算极快的场景（如缓存、简单消息推送）。

#### 模型二：多线程 Reactor（又称 Reactor + 线程池）

一个 Reactor 线程负责所有 I/O 事件的监听和分发，但它拿到请求数据后，**把具体的业务计算扔给线程池**处理。

-   **流程**：Reactor 线程负责 `accept`、读、写，以及所有 I/O 相关操作。当一个请求被完整读取后，它把请求体封装成任务，放进线程池的任务队列，让工作线程去计算（如解密、查数据库）。工作线程算完后，把结果写回给客户端，或者通知 Reactor 线程“这个 fd 可以写了”。
-   **优点**：分离了 I/O 和计算。Reactor 线程响应极快，不会被单个业务卡住；同时，多核 CPU 被线程池充分利用。
-   **缺点**：所有 I/O 操作（读、写）仍然由一个 Reactor 线程负责，在极高并发（如每秒几十万连接）下，这一个 Reactor 线程的 I/O 处理能力可能成为瓶颈。
-   **适用场景**：大多数通用网络服务。

#### 模型三：主从 Reactor（多 Reactor 多线程，Netty、Nginx、Memcached 采用）

这是目前高性能服务器的**工业标准模型**。它把“建立连接”和“处理连接上的 I/O”进一步解耦。

-   **MainReactor（主 Reactor）**：通常只有一个，**只负责 `accept` 建立新连接**。新连接到手后，用某种负载均衡算法（如轮询）分配给一个 SubReactor。
-   **SubReactor（从 Reactor）**：有多个，每个 SubReactor 和一个线程绑定，持有独立的 epoll。它负责管理分配给它的所有客户端连接的**后续所有 I/O 事件**（读、写、关闭等）。
-   **处理流程**：MainReactor 拿到新连接 -> 分给 SubReactor-1 -> SubReactor-1 把该连接的 fd 注册进自己的 epoll，之后该连接上的所有读写事件都由 SubReactor-1 处理。如果业务计算重，SubReactor 的内部还可以配合线程池。
-   **优点**：
    -   职责清晰，连接建立和 I/O 处理分开。
    -   **多核 I/O 并行**：多个 SubReactor 线程可以各自处理一部分连接的 I/O，完美利用多核 CPU，解决了单 Reactor 的 I/O 瓶颈。
    -   **高可用**：一个 SubReactor 出问题，往往不会影响其他。
-   **适用场景**：所有高并发、高吞吐的现代网络服务。

---

### 4. Reactor 与 Proactor 的区别（扩展）
-   **Reactor**：内核通知“事件就绪”（数据来了），然后用户态自己调用 `read` 把数据从内核拷贝到用户空间。这是**同步非阻塞 I/O** 的范畴。
-   **Proactor**：用户预先发起一个异步读请求（如 `aio_read`），内核不仅通知你“数据到了”，而且**内核已经把数据直接拷贝到了你提供的用户缓冲区**。当 Proactor 收到通知时，数据已经在那儿了，可以直接用。这是**异步 I/O** 的范畴。

**关系**：Reactor 是 `epoll` 的实用封装，而 Proactor 是 `io_uring` 或 AIO 的实用封装。目前主流仍是 Reactor，但 Proactor 是未来的趋势。

---

### 总结

**Reactor 模型是网络编程知识的集大成者：**

-   它用 **IO 多路复用**（epoll）解决了“一个线程等海量连接”的问题。
-   它用 **事件驱动** 将事件监听与业务处理解耦，实现高内聚低耦合。
-   它通过 **主从多线程** 解决了“CPU 密集阻塞 I/O 线程”和“单线程 I/O 处理瓶颈”的问题。

**Reactor 就是被大量工程实践验证过的、利用 epoll 实现高并发网络服务的最优解之一。**

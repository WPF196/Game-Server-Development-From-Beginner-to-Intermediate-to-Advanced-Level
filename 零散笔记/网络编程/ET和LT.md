# ET和LT
**LT 和 ET 是 Linux 内核 `epoll` 模型的两种事件触发模式，它们决定了 `epoll_wait` 在什么情况下会返回，通知你有事件发生。**

它们关注的核心是 **“状态”** 和 **“变化”** 的区别。

---

## 一、核心定义（一句话版）

- **LT（Level Trigger，水平触发）**：**“只要条件满足，就一直通知”**。只要 socket 接收缓冲区还有数据可读（或者发送缓冲区还有空间可写），`epoll_wait` 就会**持续**返回该事件，直到你把数据读完或缓冲区填满。
- **ET（Edge Trigger，边沿触发）**：**“只在状态变化时通知一次”**。只有当 socket 的状态从“不可读”变为“可读”（或者从“不可写”变为“可写”）的**瞬间**，`epoll_wait` 才返回一次。之后即使状态仍然满足，也不会再通知。

---

## 二、核心区别：一个“闹钟” vs “一个只响一次的闹钟”

我们用一个具体的例子来感受两者的差异。

### 场景复现

1.  对端一次性发来 **100 字节** 数据，全部存入内核接收缓冲区。
2.  你的程序调用 `epoll_wait` 开始监听。
3.  你的程序决定**只读取 50 字节**，剩下 50 字节留在内核缓冲区。

| 触发模式 | `epoll_wait` 的行为 | 后续会怎样？ |
| :--- | :--- | :--- |
| **LT (水平触发)** | `epoll_wait` 返回 `EPOLLIN` 事件。 | **会再次通知！** 因为缓冲区里还剩 50 字节没读，只要条件满足（有数据），`epoll_wait` 会一直返回这个事件，督促你把剩下的数据读完。 |
| **ET (边沿触发)** | `epoll_wait` 返回 `EPOLLIN` 事件。 | **不会再通知！** 因为这次通知已经是对“从无到有”这个状态变化的回应了。即使还剩 50 字节，由于状态没有再次变化（`可读→不可读→可读`），`epoll_wait` 不会再返回。除非对端**又发来新的数据**，状态再次变化，才会触发新的事件。 |

---

## 三、为什么要有这两种模式？

它们反映了不同的设计哲学：

### LT（水平触发）：**“安全、保险、易于编程”**

- **特点**：像老师反复提醒你交作业，你不交，他就一直催。
- **优点**：不容易出错。即使你一次没读完，下次循环还能接着处理，事件不会丢失。
- **缺点**：可能会产生多余的事件通知（如果你总是一次只读一点），增加一些系统调用开销。
- **muduo 的选择**：**muduo 默认使用 LT 模式**。因为陈硕（muduo 作者）认为，在大多数业务场景下，LT 的编程模型更简单、更安全，其性能损失在现代服务器上可以忽略不计。

### ET（边沿触发）：**“高效、极致、要求更高”**

- **特点**：像快递员只给你发一次“快递已到”的短信，你当天没去取，第二天他也不会再提醒你。
- **优点**：事件通知次数少，理论上可以减少不必要的 `epoll_wait` 唤醒，在高并发下 CPU 效率更高。
- **缺点**：**要求你必须一次性把数据全部处理完**（读/写到 `EAGAIN`），否则会丢失事件通知，导致数据滞留。编程复杂度高，容易出 Bug。

---

## 四、ET 模式下的“黄金法则”

如果你选择使用 ET 模式，必须遵守一个铁律：

> **对于读事件：必须循环调用 `read()`/`recv()`，直到返回 `EAGAIN` 错误，确保内核缓冲区被读空。**
> **对于写事件：必须循环调用 `write()`/`send()`，直到返回 `EAGAIN` 错误，确保内核缓冲区被写满。**

这是因为 ET 只通知一次，你必须利用这一次机会，把能做的操作全部做完。

```cpp
// ET 模式下读数据的标准写法
void handleRead() {
    while (true) {
        ssize_t n = read(fd, buffer, sizeof(buffer));
        if (n > 0) {
            // 处理数据...
        } else if (n == 0) {
            // 对端关闭连接
            break;
        } else {
            if (errno == EAGAIN || errno == EWOULDBLOCK) {
                // 数据已全部读完，退出循环
                break;
            } else {
                // 其他错误
                break;
            }
        }
    }
}
```

## 五、LT vs ET：核心差异对照表

| 对比维度 | **LT（水平触发）** | **ET（边沿触发）** |
| :--- | :--- | :--- |
| **触发条件** | 状态**满足**（如缓冲区有数据） | 状态**变化**（如从无数据到有数据） |
| **通知频率** | 持续、反复通知，直到处理完毕 | 只通知一次，即使没处理完也不再通知 |
| **编程要求** | 低，可以分批处理 | **高**，必须一次性处理干净（读到/写到 `EAGAIN`） |
| **事件丢失风险** | **无** | **有**，如果没处理完会丢失事件通知 |
| **系统调用次数** | 可能稍多（LT 可能反复唤醒） | 较少（每次状态变化只唤醒一次） |
| **适用场景** | 通用、业务逻辑复杂、追求开发效率 | 高性能、大并发、对 CPU 效率有极致要求 |
| **muduo 默认** | ✅ **使用 LT** | ❌ 不使用 |

---

## 六、总结：一个快递柜的比喻

-   **LT（水平触发）**：快递柜的灯一直亮着，直到你取走所有包裹。你随时可以取，取了一部分灯也不会灭，会一直提醒你。
-   **ET（边沿触发）**：包裹到达时，快递柜的灯闪了一下（状态变化）。无论你取没取完，灯都不会再闪。你必须趁这次亮灯的机会，把所有包裹都取走（读到 `EAGAIN`），否则剩下的包裹只能等下次新包裹到达时才能再取。

muduo 选择了 LT，就是选择了**安全、稳定、易用**，让你能更专注于业务逻辑，而不是纠结于 `EAGAIN`。这很符合 C++ 网络库“为开发者减负”的设计理念。
